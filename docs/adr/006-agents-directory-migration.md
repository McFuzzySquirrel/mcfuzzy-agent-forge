# ADR-006: Migrate from `.github/` to `.agents/` Directory

**Date:** 2026-06-17
**Status:** Accepted

---

## Context

The McFuzzy Agent Forge was originally designed solely for GitHub Copilot. Agent and skill templates were bootstrapped into `.github/agents/` and `.github/skills/` - directories that GitHub Copilot auto-detects. All template files, documentation, and the bootstrap scripts referenced these GitHub-specific paths.

Two developments prompted a migration:

1. **The Agent Skills specification** (agentskills.io) defines a harness-agnostic format for skills (directory with `SKILL.md`, `references/`, `scripts/`, `assets/` subdirectories). The spec does not mandate `.github/` as the root - skills and agents should live under a neutral directory to work with any agent runtime.

2. **Other agent harnesses** (Claude Code, Copilot CLI, future runtimes) use different detection directories (`.claude/`, `.github/`, etc.). Coupling the forge to `.github/` prevents users from targeting these harnesses without manual path rewriting.

Additionally, the `_model-launch.{sh,ps1}` wrappers generated by `forge-assign-models` Apply mode were Copilot CLI-specific band-aids. The `model:` frontmatter field is the portable mechanism; per-agent launch wrappers are unnecessary and harness-specific.

---

## Decision

### 1. Adopt `.agents/` as the Canonical Root

All forge templates, skills, and agents now reference `.agents/` as the canonical directory:

- Agents: `.agents/agents/*.agent.md`
- Skills: `.agents/skills/{name}/SKILL.md`

The forge's own repository moves its skill from `.github/skills/` to `.agents/skills/`.

### 2. Bootstrap Scripts Accept a `--harness` Flag

The bootstrap scripts now accept `--harness <preset>` to target different harnesses:

| Preset | Root Dir | Agents live at | Skills live at |
|---|---|---|---|
| `agents` (default) | `.agents/` | `.agents/agents/` | `.agents/skills/` |
| `github` | `.github/` | `.github/agents/` | `.github/skills/` |
| `claude` | `.claude/` | `.claude/agents/` | `.claude/skills/` |

Template files canonically reference `.agents/`. When a non-default harness is selected, the bootstrap script performs a find-replace (`s|.agents/|.$ROOT/|g`) during copy to adapt internal path references.

Usage:
```bash
./scripts/bootstrap.sh ~/my-project                    # default → .agents/
./scripts/bootstrap.sh ~/my-project --harness github   # → .github/
./scripts/bootstrap.sh ~/my-project --harness claude   # → .claude/
```

### 3. Remove `_model-launch` Wrappers

The `_model-launch.{sh,ps1}` wrappers generated by `forge-assign-models` Apply mode are removed. The `model:` frontmatter field is the portable, harness-agnostic mechanism for per-agent model assignment. Individual harnesses surface this field differently (VS Code honors it natively; CLI users set `COPILOT_MODEL` globally).

All references to these wrappers are purged from `forge-assign-models/SKILL.md`, `README.md`, and `docs/prompt-playbook.md`.

---

## Consequences

### Positive

- **Harness-agnostic.** Forge works with any agent runtime. Users target GitHub Copilot, Claude Code, or future harnesses via a single flag.
- **Spec-aligned.** The `.agents/skills/` directory structure matches the agentskills.io spec (skill directory with `SKILL.md`, `references/`, `scripts/`, `assets/`).
- **Cleaner `forge-assign-models`.** Removing `_model-launch` wrappers eliminates CLI-specific workarounds. The skill focuses solely on the portable `model:` and `modelFallback:` frontmatter fields.
- **Simpler mental model.** One canonical path (`.agents/`) in all templates, docs, and source code. The bootstrap script handles adaptation for specific harnesses.
- **No breaking change for existing users.** Existing bootstrapped projects keep working. New bootstraps get `.agents/` by default.

### Negative

- **Template size increases slightly** in bootstrap scripts (harness mapping + find-replace logic, ~20 extra lines per script).
- **Historical docs** (ADRs, research notes) now reference paths that differ from older versions of the project. This is expected for any migration.

### Neutral

- **Existing bootstrapped projects are unaffected.** They continue using `.github/` until re-bootstrapped. The `--harness github` flag ensures backward compatibility for users who prefer to stay on `.github/`.
- **Forge's own `.github/` directory is removed** except for any GitHub-specific configuration (CI workflows, issue templates) which live under `.github/` for GitHub platform reasons - these are unrelated to agent/skill paths.
